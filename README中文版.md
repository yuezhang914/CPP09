# ex00 — Bitcoin Exchange

## 主要目的是学什么

这一题的主要目的是学习如何使用 **有序关联容器** 处理“查表”问题。

程序需要根据一个日期查询比特币历史汇率。如果输入日期在数据库中不存在，不能随便报错，也不能使用之后的日期，而是要找到数据库中 **小于或等于该日期的最近日期**。

因此，这一题重点训练的是：

- 使用 `std::map` 保存有序的键值对；
- 理解 key-value pair 的数据组织方式；
- 使用 `lower_bound` 查找最接近的位置；
- 读取和解析 CSV 文件；
- 处理文件输入、格式校验、日期校验和数值范围校验；
- 区分“程序无法继续的错误”和“某一行输入错误”。

## 做了什么

本题实现了一个名为 `btc` 的程序。

程序启动后会先读取题目提供的 `data.csv` 文件。这个文件是比特币历史汇率数据库，格式是：

```text
date,exchange_rate
```

例如：

```text
2012-01-11,7.1
```

程序会把它保存到：

```cpp
std::map<std::string, double>
```

其中：

- `std::string` 保存日期；
- `double` 保存该日期对应的汇率。

然后程序读取用户传入的输入文件，输入文件格式是：

```text
date | value
```

例如：

```text
2011-01-03 | 3
```

程序会检查：

- 日期格式是否是 `YYYY-MM-DD`；
- 日期是否真实存在，比如不能接受 `2001-42-42`；
- value 是否是合法数字；
- value 是否为正数；
- value 是否不超过 1000。

如果一行数据合法，程序会查找对应汇率并输出：

```text
date => value = result
```

如果日期不存在，程序会通过 `std::map::lower_bound` 找到最接近的位置，并使用更早的日期对应的汇率。

## 总结学到了什么

通过这一题，我学到 `std::map` 适合处理有序查找问题。  
当数据本身有“顺序”含义时，比如日期、编号、区间边界，`map` 不只是能保存数据，还可以帮助定位最接近的数据位置。

我也进一步理解了输入校验的重要性。文件内容不能默认可信，程序需要分别检查格式、范围和逻辑合法性。尤其是日期这种数据，不能只看字符串长得像日期，还要判断月份、天数和闰年。

这一题还让我理解了 `data.csv` 的作用：它不是用户输入，而是程序运行时依赖的数据库文件。用户输入文件只提供要查询的日期和数量，真正的汇率来自 `data.csv`。

# ex01 — Reverse Polish Notation

## 主要目的是学什么

这一题的主要目的是学习如何使用 **栈结构** 解决表达式计算问题。

Reverse Polish Notation，简称 RPN，中文叫逆波兰表达式或后缀表达式。它的特点是运算符写在操作数后面。

普通表达式：

```text
3 + 4
```

RPN 表达式：

```text
3 4 +
```

这一题重点训练的是：

- 理解 RPN 的计算规则；
- 使用 `std::stack` 表达“后进先出”结构；
- 把字符串拆成 token；
- 正确处理二元运算符；
- 判断表达式是否合法；
- 在错误时输出到标准错误。

## 做了什么

本题实现了一个名为 `RPN` 的程序。

程序接收一个命令行参数，这个参数是完整的 RPN 表达式，例如：

```bash
./RPN "8 9 * 9 - 9 - 9 - 4 - 1 +"
```

程序会使用 `std::istringstream` 按空格逐个读取 token。

如果 token 是数字，就把它压入栈中。

如果 token 是运算符：

```text
+ - * /
```

程序会从栈中取出两个数字，进行计算，再把结果放回栈中。

对于减法和除法，程序特别处理了左右顺序：

- 第一次从栈里取出的数是右操作数；
- 第二次从栈里取出的数是左操作数。

所以应该计算：

```text
left - right
left / right
```

而不是反过来。

程序还会检查多种错误情况：

- 参数数量错误；
- token 非法；
- 运算符出现时栈中数字不足；
- 除以 0；
- 表达式结束后栈里不是刚好剩一个结果。

## 总结学到了什么

通过这一题，我学到 `std::stack` 非常适合处理“最近的数据先使用”的问题。RPN 的计算过程刚好符合栈的后进先出规则，因此栈比数组或链表更自然。

这一题也让我理解了 token 处理的基本思路。一个复杂字符串可以先拆成一个个小单位，再逐个判断每个单位代表数字还是操作符。

另外，二元运算符的顺序是这一题很容易出错的地方。加法和乘法看不出问题，但减法和除法必须严格区分 left 和 right。

# ex02 — PmergeMe

## 主要目的是学什么

这一题的主要目的是学习如何在不同 STL 容器上实现同一个算法，并观察容器特性对实现方式和性能的影响。

题目要求使用 **Ford-Johnson algorithm**，也叫 **merge-insert sort**，中文可以理解为归并插入排序。

这一题重点训练的是：

- 理解 Ford-Johnson 排序的基本流程；
- 使用两个不同容器实现同一排序任务；
- 比较 `std::vector` 和 `std::list` 的访问差异；
- 处理大量命令行参数；
- 统计程序处理时间；
- 不依赖 `std::sort`，而是自己实现排序逻辑。

## 做了什么

本题实现了一个名为 `PmergeMe` 的程序。

程序接收一串正整数作为参数，例如：

```bash
./PmergeMe 3 5 9 7 4
```

程序会先检查所有参数：

- 必须是数字；
- 必须是正整数；
- 不能超过 `int` 的最大范围。

然后分别使用两个容器保存并排序同一组数据：

```cpp
std::vector<int>
std::list<int>
```

排序逻辑采用 Ford-Johnson 的主要结构：

第一步，把数字两两配对。每一对中较大的数字记为 `high`，较小的数字记为 `low`。

第二步，先递归排序所有 `high`，形成一条已经有序的 main chain。

第三步，把所有 `low` 和可能存在的 stray 落单元素作为 pending 元素。

第四步，按照 Jacobsthal 相关顺序，把 pending 元素插入 main chain。

第五步，分别输出排序前、排序后，以及两个容器各自的处理耗时。

程序会输出类似：

```text
Before: 3 5 9 7 4
After: 3 4 5 7 9
Time to process a range of 5 elements with std::vector<int>: ...
Time to process a range of 5 elements with std::list<int>: ...
```

## 总结学到了什么

通过这一题，我学到同一个算法放在不同容器上，实现方式会明显不同。

`std::vector` 支持下标访问，适合二分查找和按位置计算。因此在 vector 版本中，可以更直接地通过下标定位中间位置和插入位置。

`std::list` 不支持下标访问，只能通过迭代器一步一步移动。因此在 list 版本中，很多位置操作都需要转换成迭代器操作。

这一题也让我理解了 Ford-Johnson 算法的核心思想：它不是简单地把所有数字直接排序，而是先通过配对减少问题规模，再递归排序较大值，最后按特定顺序插入较小值，从而减少比较次数。

同时，我也学到性能测试不能只看排序函数本身。题目要求时间包括数据管理和排序过程，所以从读取、转换、装入容器到排序结束都应该被计入。

# 总体总结

CPP09 的核心是 STL 容器的选择和使用。

ex00 让我学习了如何用 `std::map` 处理有序查找问题。  
ex01 让我学习了如何用 `std::stack` 处理符合后进先出规则的表达式计算。  
ex02 让我学习了如何在 `std::vector` 和 `std::list` 上实现同一个复杂排序算法，并理解不同容器对代码结构和性能的影响。

完成这个项目后，我对 STL 的理解不再只是“会用容器存数据”，而是开始理解：

- 为什么这个问题适合这个容器；
- 容器的访问方式会怎样影响算法；
- 算法要求和容器特性之间如何配合；
- 输入校验和错误处理是完整程序的一部分；
- 同样的功能，在不同数据结构上可能需要不同实现方式。
